
# 객체지향 콘솔 프로젝트 회고

## 프로젝트 개요

- **진행 기간**: 5일
- **개발 환경**: C# / .NET Console
- **프로젝트 목표**
  - C# 기본 문법(조건문, 반복문, 배열, 메서드)을 실제 코드에 적용한다.
  - 클래스를 활용하여 데이터를 구조화하고 책임을 분리한다.
  - 콘솔 환경에서 게임의 흐름(입력 → 처리 → 출력)을 직접 구현한다.

## 구현한 게임 설명

- **게임 장르**: (예: 턴제 RPG / 퍼즐 / 로그라이크 / 텍스트 어드벤처 등)
- **핵심 규칙**
  - 
  - 
- **플레이 흐름**
  1. 게임 시작
  2. 입력 처리
  3. 상태 변경
  4. 결과 출력
  5. 2~4 반복

## 설계 구조

### 주요 클래스

- `GameManager`  
  - 게임 전체 흐름을 관리
- `SceneManager`
  - Scene 단위를 보관하며, 현재 구동중인 씬의 로직 제어
- `PlayerCharacter`
  - 플레이어 상태 및 행동 관리
- `Scene`
  - 특정 씬(타이틀, 마을 등)에서의 동작 로직 구현

### 클래스 간 관계

- 어떤 클래스가 어떤 책임을 가지는지
- 서로 어떤 방식으로 협력하는지

## 사용한 C# 개념

- 클래스 / 객체
- 생성자
- 접근 제한자 (`public`, `private`)
- 제네릭 컬렉션 (`List`, `Dictionary`)
- 조건문 / 반복문
- 메서드 분리

## 잘했다고 생각하는 점

- 클래스를 역할 단위로 분리하려고 노력했다.
- 중복 코드를 줄이기 위해 메서드를 분리했다.
- 콘솔 출력 정렬이나 가독성을 고려했다.

## 아쉬웠던 점 / 어려웠던 점

- 처음 설계 없이 구현을 시작해 구조가 꼬였다.
- 입력 처리와 게임 로직이 섞여 코드가 복잡해졌다.
- 일부 기능 구현에 시간을 너무 소비해 구현하지 못한 기능이 다수 있다.

## 다시 만든다면 개선하고 싶은 점

- GameManager의 책임을 더 분리하고 싶다.
- 인터페이스를 더 활용하고 싶다.
- 상태(State) 개념을 도입해 구조를 단순화하고 싶다.

## 이번 프로젝트를 통해 배운 점

- 문법을 아는 것과 구조를 설계하는 것은 다르다는 점
- 클래스 설계가 코드 유지보수에 큰 영향을 준다는 점
- 작은 콘솔 게임이라도 구조가 중요하다는 점
